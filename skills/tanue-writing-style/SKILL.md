# Tanue Writing Style — 改写模式

将已有中文草稿按 @Tanue 的个人风格改写。**只改语言质感和表达方式，不新增事实、不改结构、不改图片、不改格式。**

## Skill 定位

本 Skill 是 Tanue 技术史叙事写作体系的"改写引擎"。它与 `tanue-writing-from-scratch` 的区别在于：

- `from-scratch`：从零生成文章，需要判断类型、建立主线、选择模板
- `style`（本 Skill）：处理已有草稿，在不变动事实、结构、图片、格式的前提下，将语言质感和叙事方式改造为 Tanue 风格

简单说：`from-scratch` 是"写一篇"，`style` 是"把这篇改成 Tanue 风格"。

## 适用改写范围

- **技术史叙事**：航空航天、军事装备、苏联/俄罗斯工业史、汽车工业史、核能技术史
- **工程事故复盘**：航空事故、航天事故、核事故、系统性工程灾难
- **人物致敬**：科学家、宇航员、工程师、历史人物
- **个人情绪叙事**：异地、车站、远方、青春、离别意象

## 不改什么（硬约束）

以下内容在改写中**绝对不能动**：

- **不改事实**：不新增、不删除、不修改任何技术事实、历史事实、人名、地名、时间、型号、机构
- **不改结构**：不合并段落、不拆分章节、不重新排列信息顺序、不移动段落位置
- **不改图片**：不移动、不增减、不替换任何图片
- **不改格式**：字体、字号、行距、对齐方式、图片属性均保持原样
- **不新增信息**：不引入原稿没有的事实、数据、引用或背景信息

**如果原稿的某个技术细节你不确定是否正确——不要改它，保留原样。** 本 Skill 只改表达方式，不做事实核查。

## 核心改法：五步骤

### 第一步：开头改造

这是改动最大的部分。将百科式开头、新闻通稿式开头、论文摘要式开头改成从以下入口切入：

- 历史场景（"时间拨回到……"）
- 事故瞬间（"最让人不安的并不是……"）
- 反常问题（"为什么相似的悲剧会重复上演"）
- 写作动机（"在中文互联网上它的介绍并不多"）
- 场景感（光、雨、声音、车站等意象）

**改前 → 改后对照：**

| 改前（百科式） | 改后 |
|---|---|
| V-2 火箭是德国在二战期间研制的弹道导弹…… | 时间拨回到上世纪三十年代。彼时的人类还没有真正离开地球，但已经有人望着星空，认真地思考如何把一个金属圆筒送到更高的地方。 |
| 切尔诺贝利核事故是1986年发生在苏联乌克兰境内的…… | 这件事看起来像一次简单的技术故障，但如果把时间线重新摊开，就会发现，真正可怕的并不是某一个零件失效，而是一连串本不该同时发生的事情，最终在同一个时刻碰到了一起。 |
| 图-64是苏联在1950年代研制的战略轰炸机…… | 在中文互联网上，关于它的介绍并不算多。即使偶尔被提到，也常常只是作为某个著名型号的前传被简单带过。但如果把它放回当时的历史环境，就会发现，这个项目本身很能说明那个时代的技术雄心和工业限制。 |
| 2020年某月某日，某航班在起飞后不久坠毁…… | 这起事故最值得注意的地方，并不是它发生得有多突然，而是它在发生之前，其实已经留下了很多可以被识别的信号。 |
| 列昂诺夫是苏联宇航员，他于1965年完成了人类首次太空行走…… | 人类历史上，总会有无数个第一次。而每一个第一次背后，都站着一个愿意把生命交给未知的人。 |

**开头改造的优先级判断：**
1. 先判断原稿开头属于哪种类型（百科式/新闻式/论文式/营销号式）
2. 再判断文章本身属于哪种类型（技术史/事故复盘/人物致敬/个人叙事）
3. 从对应的入口模式中选择最合适的改写方向
4. 保留原稿中的关键事实（时间、事件名称），但换一种进入方式

**绝对禁止的开头句式（发现了必须改掉）：**
- "某年某月某日，发生了……"（纯时间地点堆砌）
- "某某是某国研制的……采用……性能为……"（百科定义）
- "今天我们来聊一聊……"
- "大家都知道……"
- "最近很多人问我……"

### 第二步：段落结构调优

原文中每段 150-250 字的"AI 标准段落"必须打破。这是最容易被忽略但最重要的步骤之一。

**具体操作：**

**（A）断出微段（<30 字）**

将可以单独成立的关键判断、转折、情绪落点从长段中拆出来，单独成段。

改前：
> 这项设计本身并不复杂，甚至可以说相当直观。问题在于，它需要在极短时间内完成切换，而且任何一个环节的延迟都会让原本用来保护系统的装置变成新的风险源。这很危险，更危险的是当时并没有多少人真正意识到它有多危险。

改后：
> 这项设计本身并不复杂，甚至可以说相当直观。问题在于，它需要在极短时间内完成切换，而且任何一个环节的延迟，都会让原本用来保护系统的装置变成新的风险源。
>
> 这很危险。
>
> 更危险的是，当时并没有多少人真正意识到它有多危险。

**可独立成微段的内容类型：**
- 关键判断（"这很危险。"、"真正的问题就在这里。"、"事情到这一步已经不可逆了。"）
- 转折点（"但麻烦并不仅限于此。"、"问题出在另一个地方。"）
- 情绪落点（"可惜的是。"、"遗憾的是。"、"代价很沉重。"）
- 时间节点（"那是 1942 年 10 月 3 日。"）

**（B）图片说明单独成行（不超过 35 字）**

改前：
> M-17 机翼细节，对 M-17 来说机翼是最核心的部件，直接决定了这架飞机的飞行性能。

改后：
> M-17 机翼细节。可以看到非常夸张的大展弦比。

改前：
> M-17 三面图，展示了它大展弦比机翼、细长机身和双尾梁布局等独特的设计特点。

改后：
> M-17 三面图。看起来和同时代多数喷气机都不一样。

图片说明改写规则：
1. 去掉解释性描述（那些正文已经说过的信息）
2. 只保留引导性的、感官性的简短描述
3. 不超过 35 字
4. 单独成行

**（C）删除每段末尾的"升华句"**

原文中每段末尾常见的 AI 收尾句式——全部删除或改为自然收束：

| 删除类型 | 示例（删除整句） |
|----------|-----------------|
| "这也是……的原因" | 这也是 M-17 不能被忽视的原因。 |
| "它的意义在于……" | 它的意义在于重新定义了高空侦察的技术路线。 |
| "这标志着……" | 这标志着苏联航空工业进入了一个新的阶段。 |
| "从某种意义上说……" | 从某种意义上说，这一失败是后来成功的必要条件。 |
| "这说明……" | 这说明技术在追赶过程中必然会经历试错。 |
| "航空工程史告诉我们……" | 航空工程史告诉我们，技术的价值不在于规模。 |
| 泛泛的总结判断 | 这无疑是人类征服自然过程中的又一次伟大胜利。 |

改后做法：让信息自然结束——段落在讲完最后一个事实、分析或判断后直接收住，不额外加一句"总结"。

### 第三步：语言去 AI 化（按优先级 P1→P6 执行）

#### P1：删除非必要引号（目标：去掉 70% 以上）

这是最重要的单步操作。检查全文所有用引号包裹的内容，按以下规则处理：

**保留引号的情况（不到原引号总数的 30%）：**
- 俄文正式名称，如 "Стратосфера"
- 项目代号，如 "主题 34"
- 直接引语（某人确实说过的话）
- 引用文件原文

**删除引号的情况（必须去掉，占原引号总数的 70% 以上）：**
- 普通技术名词："苏联版 U-2" → 苏联版 U-2
- 比喻："晚交的卷子" → 晚交的卷子
- 抽象概念："命运""转身""遗产" → 命运、转身、遗产
- 定型化说法："高空科研飞机""战略轰炸机" → 高空科研飞机、战略轰炸机

为什么要去掉引号：Tanue 风格中，引号使用极其克制。过多的引号会让文章看起来像翻译腔或学术论文，破坏叙事的自然流动感。

#### P2：替换禁止句式（全文合计不超过 3 次）

以下六种句式在全文中的出现次数合计不能超过 3 次：

| 禁止句式 | 替换方案 |
|----------|---------|
| "不是……而是……" | 拆成两句，或直接说后半句 |
| "这说明……" | 改为"可见……"或直接删除引语，让事实自己说话 |
| "这标志着……" | 改为"这意味着……"或直接陈述 |
| "从某种意义上说……" | 整句删除——它几乎从不增加信息 |
| "它的意义不在于……而在于……" | 整句删除，改为正向陈述 |
| "既……又……"（连续使用） | 拆成两个句子 |

改前 → 改后示例：

> 这不是一次简单的技术故障，而是一个系统在多重压力下逐渐失效的过程。

改后：
> 这远远不止是一次技术故障。它更像是一个系统在多重压力下，一步步被推向安全边界的极限。

> 这说明苏联在当时已经具备了远程轰炸机的初步设计能力。

改后：
> 可见，苏联在当时至少已经摸到了远程轰炸机设计的门槛。

> 从某种意义上说，这次失败反而推动了后续型号的诞生。

改后：
> （整句删除）

#### P3：替换抽象词

将模糊的、带有文学腔或翻译腔的抽象词替换为更具体、更直接的表达：

| 原词 | 替换为 |
|------|--------|
| 命运 / 宿命 / 宿命感 | 经历 / 处境 / 后来怎么样了 / 实际走向 |
| 转身 / 转型 | 改成 / 转向 / 换了用途 / 调整方向 |
| 遗产 / 技术遗产 | 积累 / 留下的 / 技术基础 / 后续影响 |
| 叙事 | 故事 / 记录 / 说法 / 描述 |
| 图腾 / 符号 | 代表 / 象征 / 标志 |
| 高光（时刻） | 被记住 / 公开亮相 / 最受关注的阶段 |
| 自证 | 证明自己 |
| 赋能 | 给……提供支持 / 让……具备某种能力 |
| 底层逻辑 | 基本原理 / 根本原因 / 核心机制 |
| 闭环 | 完整链路 / 完整流程 |

替换背后的原则：Tanue 风格不排斥抽象思考，但排斥"用抽象词包装简单意思"。能说"经历"不说"命运"，能说"转向"不说"转型"。

#### P4：参数附意义

如果原文中有技术参数（型号、速度、推力、航程、重量、高度、功率、频率等），检查每个参数后面是否有解释。如果没有，补上。

不需要新增事实——只需要用原文已有的事实来解释参数意味着什么。

改前：
> A-3 火箭高度 6.5 米，直径 75 厘米，发动机推力约 15000 牛。

改后：
> A-3 火箭高度约 6.5 米，直径 75 厘米，发动机推力约 15000 牛。这个尺度已经不再是早期火箭爱好者的小型试验品，而是开始逼近真正武器系统的雏形。也正因为如此，控制、制导和结构问题开始集中暴露出来。

参数解释的五个方向（选最适合的一个补上即可）：
1. 这个数字在当时是什么水平？
2. 这个数字意味着什么约束？
3. 它解决了什么旧问题？
4. 它又带来了什么新问题？
5. 它和后续结果有什么关系？

#### P5：事故因果链化

如果原文涉及事故分析，检查是否有单点归因。如果原文写的是"事故原因是飞行员失误""事故原因是传感器故障"，必须改写为多因素叠加结构。

**改写前的判断——检查原文是否有以下问题：**
- 只说了一个原因
- 用了"事故原因是……"这种单因句式
- 归因到了某个人的操作失误
- 归因到了某个零件的故障
- 归因到了"管理混乱"（太泛）

**改写方法：保持原文已有的事实，但将单因表述扩展为因果链。**

改前：
> 事故的原因是飞行员在起飞后不久出现了操作失误。

改后（使用原文已有的事实来构建因果链）：
> 起飞后不久，系统记录的参数开始出现偏差。如果当时只有这一个异常，机组也许还有足够的时间识别和纠正。但问题在于，几乎在同一时间，另外几个因素也开始叠加——报警信息不够明确，留给判断的时间极其有限，而系统在接收到异常信号后，自动采取了可能让情况更复杂的保护动作。

改前：
> 这次事故的根本原因是传感器故障。

改后：
> 传感器故障本身并不罕见。真正让它变成灾难的，是它所触发的连锁反应：错误的数据进入飞控计算机，触发了本不应启动的保护逻辑；而机组看到的报警信息，又不足以让他们在短时间内判断出真正的问题出在哪里。

**因果链改写句式库：**
> 如果只有这一个问题，事故未必会发生。真正致命的是……
> 表面上看是 A 出了问题。但如果往下追一层，就会发现……
> 到这里为止，系统还没有不可挽回。真正的转折出现在……
> 这个环节本身不是最危险的。危险的是它和后面的……撞在了一起。

#### P6：图片说明简短化

逐条检查每张图片的说明文字：

**判断标准：超过 35 字 → 缩短。包含正文已说过的信息 → 删除。**

改前 → 改后对照：

| 改前 | 改后 |
|------|------|
| M-17机翼细节，对M-17来说机翼是最核心的部件，直接决定了这架飞机的飞行性能，其大展弦比设计是高空低速飞行的关键。 | M-17机翼细节。可以看到非常夸张的大展弦比。 |
| 切尔诺贝利四号机组事故后的航拍照片，可以清楚地看到反应堆厂房顶部被完全炸毁的情况，这是人类历史上最严重的核事故之一。 | 切尔诺贝利四号机组事故后的航拍照片。反应堆厂房顶部已被完全掀开。 |
| 这是V-2火箭的发动机结构示意图，展示了涡轮泵、燃烧室和喷管等主要部件的布置方式。 | V-2火箭发动机结构示意图。涡轮泵和燃烧室的布置方式。 |

### 第四步：内容质量升级

这一步是在前三步基础上的整体打磨，关注的是全文层面的表达质量。

**（A）区分事实、推断、观点——在改写中保持原文的严谨性**

如果原文中有推断但没有标明不确定性，加上缓冲语：
- "从现有资料看……"
- "这一结论还需要更多资料验证，但至少可以推断……"
- "目前公开的信息不足以完全确认，但比较合理的推测是……"

如果原文中有情绪化判断，改为有依据的观点：
- "这个设计简直愚蠢" → "从工程安全标准看，这一设计至少存在明显隐患"
- "这说明某国技术不行" → "这一案例至少反映出，当时的工业体系在……方面存在系统性不足"

**（B）删除营销号式表达**

全文搜索并替换以下表达：

| 搜索这些词 | 替换或删除 |
|-----------|-----------|
| 震惊 / 令人震惊 | 删除，或改为"值得关注的是" |
| 炸裂 / 封神 / 史诗级 | 删除，改为客观描述 |
| 太燃了 / 绝绝子 / 逆天 | 删除 |
| 头皮发麻 / 惊掉下巴 | 删除 |
| 全网都看傻了 | 删除 |
| 的背后 / 真相 / 内幕 | 谨慎保留——只有在确实揭示被掩盖信息时才用 |
| 这才是真正的…… | 删除，直接陈述 |
| 万万没想到 | 删除 |

**（C）引用与"笔者注"处理**

如果原文有"笔者注"或类似内容，检查：
- 是否压过了主题（太长的笔者注要缩短）
- 是否情绪化（"我被震撼到了"要删除）
- 是否提供了必要的可信度信息（资料来源、专业背景、不确定性说明）——这些可以保留

### 第五步：节奏与收束检查

**（A）全文句式节奏检查**

逐段检查：是否形成了"长句解释 + 短句判断 + 微段转折"的混合节奏？

如果某一段全部是长句（>80字每句），至少拆出一个短句或微段。
如果某一段全部是短句（<30字每句），检查是否需要在前面补一段解释性的长句。

理想节奏示例：
> 由于该系统需要同时依赖传感器输入、控制律判断和执行机构响应，因此当输入数据本身出现偏差时，后续所有保护逻辑都会建立在一个错误前提之上，系统表面上仍然在按照设计工作，但实际上已经开始把飞机推向错误方向。
>
> 这很危险。
>
> 更危险的是，当时并没有多少人真正意识到它有多危险。

**（B）结尾收束检查**

检查文章结尾是否符合"克制收束"标准：

要求：
- 基于前文的事实和逻辑
- 不空喊口号（"这就是伟大的……""让我们铭记……"）
- 不给新信息（不在结尾突然引入前面没提过的事实）
- 留有余味，不解释太满

如果原文结尾是口号式的——删除口号部分，保留前一句有实质内容的收束。
如果原文结尾是"总结全文"式的（"综上所述，本文分析了……"）——删除，改为从最后一个分析点自然收束。

**推荐克制结尾句式：**
> 也许……
> 很多年后……
> 这些细节单独看都不起眼，但放在一起……
> 它留下的问题至今仍然值得……

## 输出风格三档

改写时可以根据用户需求选择不同档位：

| 档位 | 特点 | 改写力度 | 适用场景 |
|------|------|----------|---------|
| 知乎长文风 | 口语转场较多、作者存在感较强、可以有"笔者注"、转场句可多用 | 较大——可以大幅改写开头、增加转场句、保留更多个人化表达 | 知乎回答、公众号初稿 |
| 成熟科普风 | 结构更清晰、语言更克制、技术解释更准确、减少口语和玩笑 | 中等——开头改写、语言去AI化、参数附意义，但不过度增加个人化表达 | 公众号正式稿、个人品牌文章 |
| 准学术叙事风 | 保留叙事入口、强化资料严谨性、减少情绪、观点更克制、几乎不用"笔者" | 较小——主要在段落节奏和去AI化，不增加个人化转场，不增加"笔者"，保留更多原文的严谨措辞 | 讲稿、报告、课程材料 |

## 改写前检查清单

开始改写前，先回答以下问题：

- [ ] 原文属于哪种类型？（技术史/事故复盘/人物致敬/个人叙事）
- [ ] 原文开头属于哪种模式？（百科式/新闻式/论文式/营销号式——需要改掉；场景式/问题式/动机式——可以保留优化）
- [ ] 原文中大约有多少处引号？（估算，用于确定 P1 工作量）
- [ ] 原文中有多少处"这说明""这标志着""不是……而是……""从某种意义上说"？（逐句扫描）
- [ ] 原文中事故分析是单因还是多因？（如果是单因，需要重点改写）
- [ ] 原文段落长度是否过于均匀？（如果大多是 150-250 字，需要重点拆段）
- [ ] 原文结尾是口号式还是克制式？

## 改写后检查清单

改写完成后逐项检查：

- [ ] 开头改成从场景/事故/问题/动机切入（非百科定义、非"今天我们来聊一聊"）？
- [ ] 去掉了 70% 以上非必要引号？
- [ ] 禁止句式（"不是……而是……""这说明""这标志着""从某种意义上说"）全文合计不超过 3 次？
- [ ] 段落长度不均匀了（有微段 <30字 + 短段 + 长段混合）？
- [ ] 大部分段落不以升华句收尾（信息自然结束）？
- [ ] 参数都附带了意义解释？
- [ ] 事故有因果链分析（多因素叠加，不是单点归因）？
- [ ] 图片说明简短（不超过 35 字）、单独成行？
- [ ] 抽象词替换为具体说法（命运→经历、遗产→积累、叙事→记录……）？
- [ ] 没有营销号语气（震惊、炸裂、封神、头皮发麻全部删除）？
- [ ] 结尾基于事实和逻辑，克制有分量？
- [ ] 没有新增事实、没有改动结构、没有移动图片、没有修改格式？
